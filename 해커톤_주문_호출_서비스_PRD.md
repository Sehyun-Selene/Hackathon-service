# 해커톤 음식 주문 & 코치 호출 서비스 PRD

- 문서 버전: v1.0
- 작성일: 2026-07-03
- 행사 일정: 2026년 9월 21일(월) ~ 22일(화), 1박 2일

---

## 1. 개요

### 1.1 배경
해커톤 참가자들이 모바일로 QR코드를 스캔해 자신의 테이블(팀)에서 음식을 주문하고, 도움이 필요할 때 코치를 호출할 수 있는 서비스. 참가자용 모바일 웹페이지와 운영진(관리자)용 웹페이지 2종으로 구성되며, 티오더(T-order)와 유사한 사용성을 목표로 한다.

### 1.2 목표
- 참가자가 별도 앱 설치나 로그인 없이 QR 스캔만으로 즉시 주문/호출 가능
- 운영진이 실시간에 가깝게 테이블별 주문 현황과 코치 호출을 파악하고 대응
- 행사 종료 후 별도 데이터 보관 없이 일회성으로 운영 가능한 경량 구조

### 1.3 규모
- 참가팀: 최대 약 125팀 (팀당 4인 기준)
- 총 참가자: 최대 500명
- 동시 접속 운영진(코치) 계정: 4명 이상

### 1.4 비목표(Out of Scope)
- 회원가입/로그인 기반 사용자 인증
- 결제 기능
- 행사 종료 후 데이터의 영구 보관 (통계 대시보드는 행사 기간 중 실시간 집계 용도로 4~5장에 포함되며, 행사 종료 후 데이터 보관과는 무관)
- 실시간 웹소켓 기반 즉시 알림 (폴링 방식으로 대체)

| 운영진(코치) | 관리자 페이지에서 주문 현황과 호출 알림을 확인/처리 |

## 3. 핵심 컨셉 및 데이터 모델

### 3.1 테이블(팀) 식별
- 팀별로 QR코드를 개별 발급하며, QR에는 테이블 번호가 URL 파라미터로 포함됨 (예: `?table=05`)
- 같은 팀원은 모두 동일한 QR을 스캔하므로 같은 테이블 번호를 공유
- 참가자는 로그인 없이, QR 스캔 즉시 자신의 테이블 주문 화면에 진입

### 3.2 인원수 및 주문 제한
- 인원수 제한 없이 자유 주문(개인당 상한 없음)을 기본으로 하되, **1인당 음식(메인) 1개 + 음료 1개**까지만 주문 가능하도록 UI에서 안내 및 유도
  - 단, 실제 인원 검증(초과 차단)은 하지 않음 — 참가자 자율에 맡기고, 관리자 화면에서 이상 수량(팀 인원 대비 과다 주문)을 육안으로 확인할 수 있도록 표시만 지원
- 메뉴는 "음식" / "음료" 두 카테고리로 구분하여 관리

### 3.3 데이터 저장 방식
- 별도 서버/DB 없이 **Claude 아티팩트 영속 저장소(window.storage, shared 모드)** 사용
- 참가자 페이지와 관리자 페이지가 동일한 shared 데이터 저장소를 읽고 씀
- 주요 키 설계 (예시):
  - `order:{table_id}` → 해당 테이블의 전체 주문 내역 (JSON: 시간대별 주문 항목 리스트)
  - `call:{table_id}` → 해당 테이블의 현재 호출 상태 (사유, 상태, 타임스탬프)
  - `call-count:{table_id}` → 해당 테이블의 누적 호출 횟수 (제한 검증용)
  - `coach-location:{coach_id}` → 코치별 최신 GPS 좌표 및 갱신 시각
- 관리자 페이지는 주기적 폴링(예: 5초 간격) + 수동 새로고침 버튼으로 최신 상태 확인
- 행사 종료 후 데이터는 별도 삭제 없이 방치 가능 (영구 보관 불필요)
- 동시 쓰기 충돌 최소화를 위해 "테이블별로 키를 분리"하는 것을 원칙으로 함 (한 팀의 쓰기가 다른 팀에 영향 없음). 단, 같은 테이블 내 동시 수정(참가자 취소 vs 관리자 상태 변경)은 드물게 마지막 쓰기가 우선 적용될 수 있음을 감수

## 4. 기능 요구사항 — 참가자 페이지

### 4.1 진입 및 테이블 인식
- QR 스캔 → URL 파라미터로 테이블 번호 자동 인식 → 별도 입력 없이 해당 테이블 주문 화면 표시

### 4.2 음식 주문
**주문 가능 시간대 (※ 아래 시각은 모두 예시/임시값이며 추후 확정 예정)**
| 구분 | 주문 가능 시간 | 식사(먹는) 시각 |
|---|---|---|
| 저녁 | 9/21 14:00 ~ 16:00 | 9/21 18:00 |
| 야식 | 9/21 17:00 ~ 19:00 | 9/21 21:00 |
| 아침 | 9/21 20:00 ~ 22:00 | 9/22 09:00 |

**화면 구조 (탭 구조 아님 — 단일 동적 화면)**
- 참가자 화면은 시간대별로 탭이 나뉘어 있지 않고, **하나의 화면이 현재 시각에 따라 자동으로 내용을 전환**하는 구조
  - 현재 시각이 주문 가능 시간대에 해당하면 → 해당 시간대의 메뉴판이 화면에 표시됨
  - 주문 가능 시간대가 아니면 → 메뉴판은 화면에서 사라지고, "다음 주문 가능 시간 안내" 문구와 함께 **코치 호출 기능만 노출**
- 주문 가능 시간대에는 마감까지 남은 시간을 카운트다운으로 표시
- 메뉴 리스트에서 "음식" / "음료" 카테고리별로 항목과 수량 선택
- 메뉴 항목마다 비건/알러지 등 식이 정보 뱃지 표시 (예: 🌱비건, ⚠️땅콩 함유 등)
- 품절된 메뉴는 선택 불가 처리(관리자가 품절 처리한 경우)
- 주문 마감 전까지 자유롭게 주문 내역 수정 및 취소 가능

**주문 내역 노출 규칙 (핵심 로직)**
- 주문 내역 조회는 시간대와 무관하게 항상 접근 가능한 영역이지만, **어떤 식사의 내역이 보이는지는 아래 규칙에 따라 자동으로 결정**됨
- 각 식사(저녁/야식/아침)의 주문 내역은 **"해당 식사의 주문 가능 시간이 시작된 시점"부터 "해당 식사를 먹는 시각"까지** 화면에 노출됨
- 이 구간들은 서로 겹칠 수 있으며, 겹치는 동안에는 **여러 식사의 주문 내역이 동시에 표시**됨 (예: 17:00~18:00에는 저녁+야식 내역이 함께 보임)
- 특정 식사를 먹는 시각이 지나면, 그 식사의 주문 내역만 화면에서 사라지고 나머지는 계속 유지됨

| 시간 구간(예시) | 화면에 보이는 주문 내역 |
|---|---|
| ~ 17:00 이전 | 저녁 |
| 17:00 ~ 18:00 | 저녁 + 야식 (겹침) |
| 18:00 ~ 20:00 | 야식 |
| 20:00 ~ 21:00 | 야식 + 아침 (겹침) |
| 21:00 ~ 09:00(9/22) | 아침 |
| 09:00(9/22) 이후 | 없음 (행사 진행에 따라 추후 결정) |

- 이 방식을 택한 이유: (1) 주문 직후 바로 본인 주문이 화면에 반영되어야 하고, (2) 식사를 하기 전까지는 이전 주문 내역도 계속 확인할 수 있어야 하기 때문 (다음 주문 가능 시간이 열리자마자 이전 식사 내역이 사라지면, 아직 그 식사를 먹기 전인데도 확인할 방법이 없어지는 문제 방지)

### 4.3 코치 호출
- "코치 호출" 버튼 클릭 → 호출 사유 선택 화면 표시
  - 사유 옵션: 멘토링 요청 / 기술 문제 / 기타 *(※ 최종 사유 목록은 추후 확정 예정 — TBD)*
- 사유 선택 후 호출 전송
- 호출 후 현재 상태를 참가자 화면에 표시: 대기중 → 처리중 → 완료
- 처리 완료 후 재호출 가능

**호출 횟수 제한**
- 팀(테이블)당 호출 가능 횟수에 제한이 있으며, 제한 횟수(예: N회, 구체적 수치는 추후 확정 — TBD)를 초과하면 시스템이 **자동으로 호출 버튼을 비활성화**하여 추가 호출을 차단
- 횟수 초과로 호출이 불가능한 경우, 참가자 화면에 "호출 가능 횟수를 모두 사용했습니다" 등 명확한 사유 안내 문구를 표시 (단순히 버튼만 비활성화되는 것이 아니라 왜 안 되는지 알려줘야 함)
- 참가자 화면에서 현재까지 사용한 호출 횟수 / 남은 횟수를 함께 표시

### 4.4 멘토(코치) 위치 확인
- 참가자 화면에서 지도 상에 표시된 멘토(코치)들의 실시간 현재 위치를 확인 가능
- 위치 정보는 GPS/좌표 기반으로 표시되며, 코치 측 기기에서 주기적으로 위치를 갱신해 공유 저장소에 반영 → 참가자/관리자 화면 모두 폴링을 통해 갱신된 위치를 확인
- 실내 행사 특성상 GPS만으로는 층/구역 구분이 어려울 수 있어, **층/구역 정보를 함께 표시**하는 방식으로 보완 (예: "3층 A구역" 같은 라벨을 좌표와 함께 노출) — 단, 실제 층/구역 목록은 행사장 정보 확정 후 반영 예정 (TBD)

## 5. 기능 요구사항 — 관리자(운영진) 페이지

### 5.1 접근
- 별도 로그인 없이 비공개 URL로 접근 (참가자에게 공유되지 않는 관리자 전용 링크)
- 접속 시 확인 중인 운영진 본인의 이름/닉네임을 간단히 선택 또는 입력 (계정 개념 없이 표시용)

### 5.2 주문 현황 확인
- 전체 테이블 리스트 및 각 테이블의 주문 내역(항목/수량) 표시
- 시간대(저녁/야식/아침)별로 필터링해서 볼 수 있음
- 메뉴별 합산 수량 자동 집계 (예: "짜장면 32개, 콜라 18개")
- 메뉴 품절 처리 기능 (품절 처리 시 참가자 화면에서 해당 메뉴 주문 불가)
- 자동 갱신(폴링) + 수동 새로고침 버튼 제공
- 최종 마감 후 전체 주문 데이터를 CSV로 내보내기 (케이터링 발주용)

### 5.3 코치 호출 알림 수신 및 처리
- 새로운 호출 발생 시 목록에 실시간(폴링 기준)으로 표시, 알림 사운드/진동으로 인지
- 호출 항목마다 테이블 번호, 호출 사유, 경과 시간 표시
- 상태 변경 기능: 대기중 → 처리중 → 완료
- 이미 다른 운영진이 "처리중"으로 변경한 호출은 중복 대응 방지를 위해 담당자 표시(누가 처리 중인지)
- 처리 완료된 호출은 목록에서 별도 영역으로 이동(과거 이력처럼 하단에 접어두기)
- 팀별 누적 호출 횟수 및 잔여 가능 횟수 표시 (횟수 제한 정책과 연동)

### 5.4 멘토(코치) 위치 공유 및 확인
- 코치가 본인 기기(관리자 페이지 또는 별도 위치 공유 화면)에서 위치 공유를 활성화하면, 브라우저 GPS 권한을 통해 현재 좌표를 주기적으로 공유 저장소에 전송
- 관리자 페이지에서 전체 코치의 실시간 위치를 지도 위에서 확인 가능 (호출 배정/대응 판단에 활용)
- 참가자 페이지에서도 동일한 지도 화면으로 코치 위치 확인 가능 (4.4 참조)

### 5.5 주문/호출 이력 통계 대시보드
- 행사 기간 전체 또는 시간대별 주문/호출 통계를 한눈에 확인
- 주문 통계: 시간대별 총 주문 건수, 메뉴별 누적 주문량, 테이블별 주문 패턴
- 호출 통계: 팀별 호출 횟수, 호출 사유별 분포, 평균 처리 소요 시간(호출~완료), 처리 담당 코치별 처리 건수
- 팀별 호출 횟수 제한과 연계해 "제한에 근접/도달한 팀" 하이라이트 표시

## 6. 비기능 요구사항

| 항목 | 요구사항 |
|---|---|
| 플랫폼 | 모바일 웹 (반응형), 별도 앱 설치 불필요 |
| 인증 | 참가자: 인증 없음(QR로 테이블 식별) / 관리자: 비공개 링크 + 이름 표시만 |
| 데이터 저장 | 서버/DB 없이 아티팩트 영속 저장소(shared) 사용, 행사 후 보관 불필요 |
| 갱신 방식 | 자동 폴링(수 초 간격) + 수동 새로고침 버튼 |
| 동시성 | 테이블 단위로 데이터 키 분리하여 충돌 최소화 |
| 성능 | 최대 125팀(약 500명) 동시 사용 가능한 수준의 폴링 주기 및 UI 반응성 |
| 접근성 | 큰 버튼, 명확한 색상 대비, 모바일 터치 최적화 |

## 7. 화면 흐름 (User Flow 요약)

**참가자**
1. QR 스캔 → 테이블 번호 자동 인식된 랜딩 화면
2. 현재 시각이 주문 가능 시간대면 메뉴판(음식/음료) 자동 노출 → 선택 → 수량 조절 → 제출 / 주문 가능 시간대가 아니면 메뉴판 대신 "다음 주문 가능 시간" 안내만 표시
3. 본인 팀의 주문 내역은 시간대와 무관하게 화면 내에서 상시 확인/수정/취소 가능 (마감 전까지)
4. "코치 호출" 버튼: 호출 사유 선택 → 호출 전송 → 처리 상태 확인 (화면 내 항상 노출되는 기능)

**운영진**
1. 비공개 링크 접속 → 이름 선택/입력
2. [주문 현황] 탭: 테이블별/메뉴별 집계 확인, 품절 처리, CSV 내보내기
3. [호출 알림] 탭: 신규 호출 확인(사운드) → 처리중 전환 → 완료 처리

## 8. 기술 구현 방향

- **참가자 페이지와 관리자 페이지는 각각 독립된 별도의 웹 애플리케이션(React 기반)으로 제작**되며, 배포 URL도 서로 다름. 두 페이지는 코드/화면 구성이 완전히 분리되어 있고, 오직 **shared 모드의 `window.storage` 데이터를 통해서만 간접적으로 연동**됨 (직접적인 API 호출이나 실시간 연결은 없음)
- 데이터 영속화는 `window.storage`의 **shared 모드**를 사용해 두 페이지 간 데이터 공유
- 시간대별 주문 가능 여부는 클라이언트의 현재 시각과 사전 정의된 일정(9/21~22 스케줄)을 비교하여 판단
- 각 테이블 QR은 동일한 배포 URL + `?table=` 파라미터로 구성, 사전에 최대 약 125개 테이블 번호에 대한 QR 이미지를 일괄 생성

## 9. 리스크 및 고려사항

- **폴링 방식의 한계**: 완전한 실시간이 아니므로, 코치 호출처럼 긴급한 알림은 폴링 주기를 짧게(예: 3~5초) 설정할 필요
- **인원수 초과 주문 검증 부재**: 시스템이 강제로 차단하지 않으므로, 팀별 과다 주문은 관리자 화면에서 육안 확인에 의존 → 필요 시 추후 "팀 인원수 입력" 기능 추가 검토 가능
- **동시 쓰기 충돌**: 같은 테이블 내에서 참가자와 관리자가 동시에 같은 데이터를 수정할 경우 마지막 쓰기가 우선됨 → 실사용 빈도상 낮은 리스크로 판단
- **관리자 링크 노출 방지**: 비공개 링크가 참가자에게 유출되지 않도록 배포/공지 방식에 유의 필요
- **실내 GPS 정확도 한계**: 해커톤이 실내에서 진행될 경우 GPS 좌표만으로는 층/구역 구분이 어려울 수 있음 → **층/구역 정보를 함께 표시하는 보완 방식을 적용하기로 결정**. 단, 실제 행사장의 층/구역 구성 정보는 아직 확정되지 않아 추후 확보되는 대로 반영 필요 (TBD: 층별 명칭/구역 리스트, 코치 위치 입력 시 층/구역 선택 UI 설계)
- **코치 위치 공유 동의**: 코치 본인 기기의 위치 공유 권한(GPS) 허용이 필요하므로, 사전에 코치 대상 안내 필요
