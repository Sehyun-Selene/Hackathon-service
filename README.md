# 해커톤 음식 주문 & 코치 호출 서비스

PRD(`해커톤_주문_호출_서비스_PRD.md`) 기반 구현. **두 개의 완전히 독립된 웹 앱**으로 구성됩니다.

```
해커톤 티오더/
├─ participant-app/   참가자용 모바일 웹 (QR 스캔 → 주문/호출)  — 독립 프로젝트
├─ admin-app/         운영진(관리자)용 웹 (주문 현황/호출 처리) — 독립 프로젝트
├─ shared-api/        두 앱이 공유하는 초경량 KV API (의존성 없음) — 실제 배포 시 필요
├─ preview-server.js  로컬 연동 테스트용 정적 서버 (의존성 없음)
└─ README.md
```

두 앱은 코드/빌드/배포가 완전히 분리되어 있고, **오직 공유 데이터 저장소를 통해서만** 간접적으로 연동됩니다 (직접 API 호출 없음).

> ⚠️ **아티팩트 shared storage 관련 정정**: 처음엔 PRD 3.3의 "Claude 아티팩트 영속 저장소(window.storage, shared 모드)"를 그대로 쓰려 했으나, 실제로 아티팩트를 발행해 검증한 결과 이 환경에는 `window.storage` API 자체가 없었습니다 (지원 capability 목록에 storage 없음 — downloads/mcp/self만 있음). 그래서 `shared-api/`라는 초소형 API 서버를 대신 두었고, 키 설계(3.3)는 그대로 유지했습니다.

## 🗄️ 공유 저장소 & 키 설계 (PRD 3.3)

`src/lib/storage.js` 어댑터가 저장소를 추상화합니다.

- **`API_BASE_URL`이 설정된 경우**: `shared-api/`(작은 공유 KV 서버)를 호출 — 참가자/관리자 앱이 서로 다른 배포 URL(다른 origin)이어도 실제로 데이터가 공유됨. **실제 행사 배포 시 반드시 이 방식을 사용하세요.**
- **비어 있는 경우**: `localStorage` 폴백 (`hackathon-torder:` 접두사) — 같은 브라우저 안에서만 유효. 로컬 개발/시연용으로만 사용 (실제 참가자 500명은 각자 다른 기기라 이 모드로는 연동되지 않음).

| 키 | 내용 | 쓰는 쪽 |
|---|---|---|
| `team:{teamId}` | 팀 정보(인원수·알러지) | 참가자(등록) |
| `team-roster` | 등록된 팀 id 목록 (관리자 열거용) | 참가자 |
| `order:{teamId}` | 팀의 식사별 주문 내역 | 참가자 |
| `call:{teamId}` | 호출 목록(사유/상태/타임스탬프/담당코치) | 참가자(생성), 관리자(상태 변경) |
| `call-count:{teamId}` | 누적 호출 횟수 (제한 검증) | 참가자 |
| `coach-roster` | 등록 코치 목록 `{coaches: [{id, name, groupId, group}]}` | 코치(관리자 앱) |
| `soldout` | 품절 메뉴 map | 관리자 |

팀별 키 분리로 동시 쓰기 충돌을 최소화합니다.

## 👥 팀 등록 흐름 (공통 QR)

모든 팀이 **같은 QR/URL**을 씁니다. 참가자가 처음 접속하면 팀 정보를 직접 입력합니다.

1. QR 스캔 → 팀 등록 화면 (팀 번호 · 인원수 · 알러지) — 계열사는 선택하지 않음
2. 입력 후 시작 → 이 기기에 팀 정보가 기억되어 재접속 시 바로 주문 화면
3. 같은 팀원은 **같은 팀 번호**를 선택 (이미 등록된 번호면 정보가 자동 채워짐)

- **음식 주문**은 등록한 **인원수만큼만** 담을 수 있음 (음료는 제한 없음)
- **알러지**는 미리 대응 메뉴를 두는 게 아니라, 수집된 현황을 **관리자가 주문현황 탭의 "알러지" 패널에서 확인**해 케이터링 대체 메뉴를 준비하는 용도
  - **사람 단위로 구분 입력**: "인원 추가" 블록마다 한 사람의 알러지를 모아 선택 → 1명이 여러 개인지, 여러 명이 각각 하나씩인지가 그대로 구분되어 저장됨 (`team.allergies = [[사람1 알러지...], [사람2 알러지...]]`). 관리자 화면엔 "1인(땅콩·견과류), 2인(우유)"처럼 표시되어 대체 메뉴를 몇 개·어떤 조합으로 준비할지 바로 판단 가능

## 🧑‍🏫 코치 담당 구조 (팀 번호 기준 배정)

계열사 대신, **팀 번호로 코치를 배정**합니다. `config.COACH_GROUPS`에 조(1~4조)별로 담당할 팀 번호를 채워두면(예: `teamNumbers: [1, 2, 3]`), 그 팀들의 호출은 해당 조가 우선 응답합니다. **지금은 팀 번호가 비어 있어(TBD) 전부 미배정 상태** — 팀 편성이 확정되면 숫자만 채우면 됩니다.

- 코치는 입장 시 이름 + **담당 조** 선택
- 참가자 화면엔 자기 팀 번호가 배정된 조가 있으면 "OO조 담당 코치가 응답합니다"로 안내, 없으면(TBD 상태) 일반 안내 문구만 표시
- 호출 알림에서 우리 조 담당 팀 호출이 상단에 강조 표시되고, "우리 조만" 필터 제공
- **코치 현황 탭**: 조별 코치 리스트 + 각 코치 상태(🟢 대기 / 🔴 팀 XX 대응 중). 특정 조 코치가 전원 대응 중(⚠️)이면 다른 조 코치가 지원하도록 판단하는 용도
- **호출별 횟수 포함 여부는 처리하는 관리자가 직접 결정**: 호출 알림 탭의 각 호출 카드에 "횟수 포함" 체크박스가 있어, 시설 문제처럼 "도움 요청"과 성격이 다른 호출은 체크 해제해 팀의 누적 호출 횟수에서 제외할 수 있음 (참가자 화면에는 "(횟수 미포함)"으로 표시됨)
  - ※ 실내 GPS 정확도 한계 때문에 위치 지도 방식은 사용하지 않고, 이 코치 현황 리스트로 대체했습니다.

## 🚀 실제 배포 순서 (행사에서 실제로 쓰려면 이 순서대로)

### 1) 공유 API 서버 배포 (Render 기준)

1. [render.com](https://render.com) 가입 → New → Web Service
2. 이 저장소의 `shared-api` 폴더를 연결 (Root Directory: `shared-api`)
3. Build Command: 비워둠 / Start Command: `node server.js`
4. 배포 후 발급되는 주소 확인 (예: `https://hackathon-torder-api.onrender.com`)
5. `https://<그 주소>/health` 접속해서 `{"ok":true,...}` 뜨는지 확인

### 2) 두 앱의 config.js에 API 주소 반영

`participant-app/src/config.js`, `admin-app/src/config.js` **양쪽 다** 아래 값을 1)에서 받은 주소로 변경:

```js
export const API_BASE_URL = 'https://hackathon-torder-api.onrender.com'
```

### 3) 참가자 앱 / 관리자 앱을 각각 배포

Vercel/Netlify 등에 `participant-app`, `admin-app`을 **각각 별도 프로젝트**로 배포 (build: `npm run build`, output: `dist`). 두 앱 모두 1)의 같은 API 서버를 바라보므로, 서로 다른 도메인이어도 데이터가 공유됩니다.

> ⚠️ Render 무료 티어는 일정 시간 무요청 시 슬립 상태가 되어, 첫 요청 응답이 느릴 수 있습니다. 행사 시작 전에 `/health`로 미리 한 번 깨워두세요.

## 🧪 로컬 개발/연동 테스트

Node.js 18+ 필요.

```powershell
# 1) 공유 API 서버 (터미널 1)
cd shared-api ; npm run start   # http://localhost:3001

# 2) 참가자/관리자 앱 — 각 config.js의 API_BASE_URL을 http://localhost:3001 로 설정 후
cd participant-app ; npm install ; npm run dev   # http://localhost:5173/
cd admin-app       ; npm install ; npm run dev   # http://localhost:5174/
```

`API_BASE_URL`을 채우면 5173/5174처럼 **포트(origin)가 달라도** shared-api를 통해 실제로 데이터가 공유됩니다 (Vercel에 각각 배포했을 때와 동일한 상황을 로컬에서 재현). `API_BASE_URL`을 비워두면 `localStorage` 폴백만 쓰므로, 같은 브라우저의 두 탭이라도 무관하게 동작 — 이 경우는 아래처럼 같은 origin으로 묶어서 봐야 합니다.

```powershell
# API 서버 없이 UI만 빠르게 훑어볼 때 (같은 origin으로 묶어서 localStorage 공유)
cd participant-app ; npm run build
cd ../admin-app    ; npm run build
cd ..              ; node preview-server.js
# → http://localhost:4173/participant/
# → http://localhost:4173/admin/
```

## 🕐 시간 시뮬레이션 (개발/시연용)

행사 일정(2026-09-21~22) 밖에서도 테스트할 수 있도록 URL에 `?now=` 파라미터를 지원합니다.

```
/participant/?now=2026-09-21T14:30   ← 저녁 주문 시간대로 시뮬레이션
/admin/?now=2026-09-21T17:30         ← 저녁+야식 내역 겹침 구간
```

오프셋 방식이라 카운트다운/시간 흐름도 정상 동작합니다. 파라미터가 없으면 실제 현재 시각을 사용합니다.

## 📱 QR 코드

**위 "실제 배포 순서"를 마친 뒤**, 참가자 앱 배포 URL **하나**로 공통 QR을 만들면 됩니다 (팀별로 다르게 만들 필요 없음).

## 구현된 기능 요약

**참가자**: 공통 QR → 팀 등록(팀번호·인원수·알러지, 계열사 선택 없음) · 이용 안내 섹션 · 시간대 자동 전환 단일 화면 · 마감 카운트다운 · 음식/음료 카테고리 + 식이 뱃지 + 사진 자리 · **인원수만큼 음식 수량 제한** · 품절 반영 · 주문 수정/취소 + 하단 고정 요약바 · 주문 내역 노출 규칙(orderStart~eatAt, 겹침 표시) · 코치 호출(사유 선택, 팀 번호 기준 담당 조 안내, 상태 표시 + 진동, 횟수 제한) · 시간대별 다크모드

**관리자**: 이름 + 담당 조 입장 · 팀별 주문(담당 조·인원수·과다 표시) + 시간대 필터 + **팀 번호 검색** · 메뉴별 합산 집계 · **알러지 현황 패널** · 품절 토글 · CSV 내보내기(담당 조·인원수 포함) · 호출 알림(사운드/진동, 담당 조 강조 + "우리 조만" 필터, 대기중→처리중→완료, 예상 처리시간) · **코치 현황(조별 대기/대응 상태, 전원 대응 시 경고)** · 통계 대시보드 · 자동 폴링 + 수동 새로고침

**공통**: 서로 다른 origin 간 `shared-api` 통한 실시간 연동 · 서버 통신 실패 시 무음 폴백이 아닌 명확한 오류 알림/배너
